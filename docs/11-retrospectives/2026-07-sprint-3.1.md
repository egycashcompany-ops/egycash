# Sprint Retrospective — Sprint 3.1 (Release v0.3.0)

**Sprint:** 3.1 · **Capability:** File Management Service · **PR:**
[#6](https://github.com/egycashcompany-ops/egycash/pull/6) · **Merged:** 2026-07-09 ·
**Outcome:** ✅ Delivered, architecture review **Implementation Approved**

## 1. Sprint goal

Deliver the platform **File Management Service** (ADR-010, Platform Core §7) as a single
capability (BD-006): generic, reusable by every future module, with pluggable storage
providers, a complete file lifecycle, category-driven security rules, and extension points
for scanning/OCR/thumbnails — proving that a Layer-1 capability can ship end-to-end on the
Sprint 2.1 platform foundations.

## 2. Delivered features

- **Five storage providers** behind one `StorageProvider` interface: Local disk, Railway
  volume, Amazon S3, MinIO (S3-compatible), Azure Blob — selection fully
  configuration-driven (`STORAGE_DRIVER`), validated at boot.
- **Lifecycle:** multipart upload → authorized/audited download → replace-as-new-version
  (groups, history retrievable) → archive/restore → soft delete → permanent delete
  (`file.purge`, break-glass, audit keeps the fingerprint).
- **Metadata:** original/stored filename, MIME, extension, immutable `sha256:` checksum,
  size, upload timestamp, uploader, entity reference, category, tags, visibility, scan
  status, storage driver+key.
- **Security:** per-route `authorize()`; visibility-aware download authorization (private ⇒
  `file.download`); **signed-URL abstraction** (native S3/MinIO presigning, Azure SAS,
  app-level HMAC capability URLs for disk drivers); per-category MIME/size rules;
  traversal-safe keys; blocked files undownloadable; every download audited.
- **Extension points:** `registerFileProcessor` seam (virus scan / OCR / thumbnail) running
  in the worker on the new `files` queue; seven `platform.file.*` events on the reliable
  tier with versioned payloads.
- **Docs:** [files-service.md](../02-architecture/files-service.md) (API reference, provider
  matrix, upload/download sequence diagrams); DB schema + permission matrix regenerated.

## 5. Architecture review summary

Reviewed against the full checklist (architecture, storage, security, metadata,
performance, events, extension points, future compatibility, testing, documentation) —
**Implementation Approved**, no service-code changes required. Highlights: boundaries and

## 6. Lessons learned

1. **Blanket gitignore patterns are landmines.** `storage/` matched a *source* directory;
   local gates passed because files existed on disk, so only CI caught it. Anchor ignore
   patterns to paths (`/storage/`), and treat "works locally, fails in CI" as a packaging
   suspect first.
2. **`git status` before push is not enough** — a pre-push check that staged output matches
   the compiled input set (or simply `git clean -ndX` review when adding new top-level
   directories) would have caught this earlier.
3. Single-capability sprints (BD-006) worked: scope stayed tight, review surface was
   readable, and the platform seams (audit, events, queues, permissions) absorbed a new
   Layer-1 service with zero changes to existing services.
4. Deprecated dependencies surface late: multer 1.x was pulled first and flagged only by
   `npm install` warnings — checking advisory status *before* choosing a version is cheaper.

## 8. Risks

- **Presigned cloud URLs cannot be revoked** once issued — a delete/block within the TTL
  window (default 300 s) does not invalidate an already-issued cloud URL. Mitigation: short
  TTL; app-signed URLs (disk drivers) re-check state at stream time.
- **Cloud drivers are configuration-tested, not live-tested** — first production use of
  S3/MinIO/Azure must be preceded by a staging smoke with real credentials.
- Driver migration requires the ADR-010 data-move script before cutover (see debt item 4).

## 9. Follow-up recommendations

1. Staging smoke against a real bucket before any cloud driver goes live.
2. Bundle the three small hardenings (pending-scan policy, transactional lifecycle events,
   missing test cases) into the next files-touching sprint rather than a dedicated one.
3. Take the Sentry major bump as a standalone chore PR.
4. When CDN/object-storage migration is scheduled, add the per-file driver registry first.
5. Keep BD-006 sizing for phase 2.2's remaining capabilities: sequences, notifications,
   localization — each its own sprint.
