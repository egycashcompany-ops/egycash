## 1. Sprint goal

Deliver **Stage 1 (Applicants)** of the approved seven-stage recruitment workflow as the
platform's **first Layer 2 business module** — proving the module manifest system
(registration, permissions, routes, collections, seeds, events) end-to-end on a real
business capability, backend-first (OQ-29), with every unbuilt dependency behind a
swappable abstraction (OQ-30), strictly within the frozen
[Sprint 4.1 plan](../12-planning/sprint-4.1-plan.md).

## 2. Delivered features

- **The `hr` module** — first consumer of the module plugin system: manifest-declared
  permissions (`applicant.{view,create,edit,delete,export}`, `applicant.verifyIdentity`,
  `applicantSource.manage`), routes under `/api/v1/hr`, module-prefixed collections
  (`hr_applicants`, `hr_applicant_sources`, `hr_sequences`), boot-time seed of the 10
  approved applicant sources, and `hr.applicant.*` events. Two small **additive platform
  seams** made this possible within the layer boundary: the `platform/web` HTTP-helper
  barrel and `manifest.seed` wiring in the boot sequence.
- **Requisition-driven intake pipeline** (BD-001): one `register()` entry point for every
  channel; mandatory, immutable Job Requisition reference validated behind the Stage-0
  seam (structural-only until the Requisition capability lands).
- **Registration**: manual, National-ID-derived, and ID-less (identity-unverified) paths;
  deterministic Egyptian National-ID parsing (birth date / gender / governorate derived
  from the 14-digit number, checksum-structural validation) — real, not stubbed;
  live-uniqueness enforced by a partial unique index; masked-by-default DTOs.
- **OCR seam** (OQ-30): provider interface + null stub; extraction never blocks or gates
  registration; derivation applies identically to OCR'd and typed numbers.
- **Organization-wide applicant numbering** `APP-{YYYY}-{seq:6}` (BD-002) via an atomic
  module-local counter, yearly reset, unique-index second line of defence.
- **Duplicate detection** (heuristic: live National-ID / normalized phone / normalized
  name+birth-date) — flags for recruiter resolution, never auto-blocks.
- **Lifecycle**: audited identity verification (human confirmation, §2.1 rule 4),
  audited withdrawal (idempotent), optimistic-concurrency updates.
- **List/search/export**: filterable/sortable/paginated list; **Arabic-normalized
  search** (hamza/alef/taa-marbuta folding) over a denormalized `searchName`; audited,
  PII-masked (unmask permission-gated) CSV export; generic per-row-audited bulk executor
  (bulk-withdraw only, per OQ-27's deferral).
- **Attachments** delegated to the platform Files service (title + category + notes,
  applicant-scoped endpoints, transactional count).
- **Docs:** [recruitment-applicants.md](../02-architecture/recruitment-applicants.md).
  The 7 new `hr` permissions are declared in the module manifest and synced to the DB
  registry at boot (ADR-004); the *generated* permission-matrix document covers the
  platform code catalog only — see follow-up 4 below.

## 5. Architecture review summary

Self-review against the module-structure checklist plus EGYCASH's independent review —
**approved, no blocking findings**. Highlights: the module boundary holds (no
`infrastructure` imports from `modules/`; platform access via `platform/*` barrels only);
deleting `modules/hr/` would leave the platform compiling; collections carry the module
prefix; every mutation audited; abstractions (OCR, requisition validator) are true seams
with safe defaults.

## 6. Follow-ups recorded (EGYCASH review — non-blocking, logged for future sprints)

1. **Concurrency testing for applicant numbering** — the allocation is a single atomic
   `$inc` and the unique index is the backstop, but a dedicated parallel-registration
   stress test would prove the no-gap/no-collision property under load.
2. **Document the duplicate-detection heuristic** in the architecture reference in more
   depth (exact probe fields, normalization applied, flag-resolution workflow) so
   recruiters' expectations match the implementation.
3. **Search optimizations** — the regex-over-`searchName` approach is index-assisted for
   prefix patterns only; if applicant volume grows, evaluate a text index or n-gram
   strategy for contains-style Arabic search.
4. **Extend `gen-permission-matrix.mjs` to include module-manifest permissions**, so the
   generated matrix document shows the whole runtime registry (platform catalog + modules)
   instead of the platform catalog alone.

## 7. Lessons learned

1. **The module manifest system paid off on first contact** — the kernel needed only two
   additive seams (`platform/web`, seed wiring) to host a full business module; no
   existing service changed behavior.
2. **Layer rules caught real design pressure early**: the ESLint boundary (modules must
   not import `infrastructure`) forced the HTTP-helper barrel decision *before* code
   spread, instead of after a refactor.
3. **Abstractions-with-safe-defaults (OQ-30) kept the sprint unblocked** — OCR and the
   Stage-0 requisition dependency would otherwise have stalled Stage 1; the seams cost
   little and the null/permissive defaults are honest about what exists.
4. **Deriving identity fields from the National ID server-side** (never trusting client
   values) eliminated a whole class of data-consistency questions at intake — worth
   repeating for any future self-describing identifier.
