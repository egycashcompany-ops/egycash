## 1. Sprint goals

Extend the `hr` Recruitment module — the platform's first Layer 2 business module (Stage 1,
Sprint 4.1) — with the next two stages of the **EGYCASH-approved recruitment workflow**,
strictly one stage at a time, additive to what came before, and touching no later stage:

- **Stage 2 — Initial Screening** (v0.7.0): decide a registered applicant to a single
  terminal outcome, Accepted or Rejected (OQ-32).
- **Stage 3 — Interviews** (v0.8.0): run the administrator-configurable interview rounds
  (OQ-31) with a panel, per-interviewer evaluations, and a gated pass/fail decision that
  progresses the applicant.

## 2. Delivered features

### Stage 2 — Initial Screening (`hr_screenings`)

- **One screening per applicant** (partial unique index), lifecycle `pending →
  accepted | rejected`. Per OQ-32 there are **only two outcomes**; "needs more information"
  is not a state — it is a note appended to a screening that stays `pending`. Screening notes
  and the **mandatory rejection reason** are stored.
- **Pipeline wiring:** a rejection transitions the applicant to the terminal `rejected`
  status (which frees the live National-ID for a fresh application, exactly like a
  withdrawal); an acceptance leaves the applicant live for interviews.
- Permissions `screening.{view,create,edit,decide}` — **`decide` is a separate grant from
  `edit`** (edit only appends notes). Events `hr.screening.{created,decided}`; the applicant
  lifecycle gained the terminal `rejected` status and `hr.applicant.rejected`.

### Stage 3 — Interviews (`hr_interviews`, `hr_interview_stages`)

- **Administrator-configurable interview stages** (OQ-31): an ordered, localized,
  deactivatable catalog seeded with the two default rounds; number/names/order are
  admin-managed thereafter (`interviewStage.manage`).
- **Interview aggregate** with a unified **panel** — each member carries an individual
  evaluation state (`pending` / `submitted` / `skipped`). Lifecycle: schedule → reschedule
  (date/time only) · **reassign panel** (independent of the schedule) · cancel ·
  per-interviewer evaluate · decide.
- **Workflow gate & progression:** the earliest stage requires a passed screening, each later
  stage the previous stage passed, one live interview per stage; the decision is **blocked
  until every panel member is `submitted` or `skipped`**; passing the final stage clears the
  interview phase, failing any round rejects the applicant.
- **Notifications integration:** schedule / reschedule / cancel notify the panel through the
  platform Notifications service, fire-and-forget (never blocking). The HR seed registers the
  three interview templates at boot via a new additive seam — `notificationTemplateService`
  exposed on the `platform/notifications` barrel.
- Permissions `interview.{view,create,edit,cancel,evaluate,decide}` + `interviewStage.manage`;
  events `hr.interview.{scheduled,rescheduled,cancelled,evaluated,decided}`; the
  `hr.applicant.rejected` payload was made source-agnostic (screening or interview).

## 5. Architecture review summary

Both stages self-reviewed against the module-structure checklist plus EGYCASH's independent
review — **approved, no blocking findings**. Stage 3 review raised two improvements, both
implemented before merge:

1. **Panel reassignment decoupled from rescheduling** — a dedicated
   `POST /hr/interviews/:id/panel` (`interview.edit`) changes the panel without touching the
   schedule; retained members keep their state, added members start pending and are notified,
   removed members drop off.
2. **Decision gated on per-interviewer completion** — the unified panel model gives each
   member a `pending`/`submitted`/`skipped` state; `POST /hr/interviews/:id/panel/skip` marks
   an interviewer absent/excused; a decision is blocked while anyone is still `pending`.

The module boundary continued to hold (no `infrastructure` imports from `modules/`; platform
access via `platform/*` barrels only; cross-feature access to Applicant/Screening via their
`index.ts` barrels); every mutation — schedule, reschedule, reassign, cancel, evaluate, skip,
decide — is audited; collections carry the `hr_` prefix.

## 6. Follow-ups recorded (EGYCASH review — non-blocking, approved to remain as designed)

The following were confirmed as **intentional scope decisions** for the current stages, and
are candidates for future sprints if the business calls for them:

1. **Interview stages are global**, not per-Job-Requisition. A per-requisition/per-job
   interview template model is a larger future capability.
2. **Fixed evaluation model** (recommendation + optional rating + notes) rather than
   configurable evaluation forms per stage/job (a form-builder is a separate feature).
3. **Manual final decision** with no computed aggregate/weighted interview score;
   per-interviewer recommendations inform the decider but are not auto-aggregated.
4. Standing Stage-1 backlog items remain open (applicant-numbering concurrency test, deeper
   duplicate-heuristic docs, Arabic contains-search optimization, extending
   `gen-permission-matrix.mjs` to include module-manifest permissions).

## 7. Lessons learned

1. **The one-stage-per-sprint discipline paid off.** Each stage plugged into the previous one
   through the aggregates' public barrels and the applicant lifecycle, with no rewrites — the
   screening→interview gate and the rejection transitions are thin, explicit seams.
2. **A unified panel model beat parallel arrays.** Collapsing "roster + evaluations" into one
   `panel` array with a per-member state made the decision gate a one-line invariant and
   removed a whole class of "which interviewer submitted?" bookkeeping — a direct result of
   the review's second improvement.
3. **Schema defaults can hide dead code.** The interview-stage sort bug was a reminder that a
   `?? fallback` over a value the Zod schema already defaults is never reached; default-aware
   sorting needs to key off an explicit caller signal.
4. **Notifications stayed a true seam.** The module registers its own templates at boot and
   calls `notify()` fire-and-forget; a notification failure can never fail a recruitment
   operation, consistent with the audit-never-block principle.
