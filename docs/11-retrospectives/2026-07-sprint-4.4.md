## 1. Sprint goal

Deliver **Stage 4 (Job Offer)** of the approved seven-stage recruitment workflow: extend an
applicant who cleared all interview rounds a versioned compensation offer, drafted → sent →
accepted / rejected / expired / withdrawn, with automatic expiration and notifications —
strictly Stage 4, additive, touching no later stage.

## 2. Delivered features

- **Job Offer aggregate** (`hr_job_offers`) with the full package: salary (`Money`),
  allowances, benefits, job title, department, branch, reporting manager, employment type,
  probation period, start date, offer validity, and notes.
- **Lifecycle** `draft → sent → accepted / rejected / expired / withdrawn` with **version
  history** (every revise snapshots the prior package) and the **one-active-offer-per-
  applicant** invariant (partial unique index + service check).
- **Immutable, human-readable offer number** `JO-{YYYY}-{seq:6}` — organization-wide, atomic
  per-year counter in the shared `hr_sequences` collection, unique-index backstop; searchable
  via a list filter. HR references offers by this number, not the ObjectId.
- **Immutable accepted-revision snapshot** — acceptance freezes the exact accepted terms and
  their revision number; Employee Creation (Stage 5) consumes this, decoupled from the live
  offer.
- **Guards**: interview-completion gate on creation; a future validity required to send; a
  lapsed sent offer cannot be accepted; an applicant who already accepted cannot be re-offered;
  the accepted-offer gate (`acceptedOfferFor`) exposed for Stage 5.
- **Automatic expiration**: a scheduled sweep (`hr.jobOffers.expire`) flips sent offers past
  validity to `expired`. Enabled by a new **additive kernel seam** — `ModuleManifest.
  scheduledTasks`, the module analogue of the `seed`/`eventSubscriptions` seams.
- **Notifications** for offer sent / accepted / rejected / expired (fire-and-forget, to the
  hiring manager + the offer's author); **full audit trail** on every transition. Permissions
  `jobOffer.{view,create,edit,send,respond,withdraw}`; events `hr.jobOffer.*`.

## 5. Architecture review summary

Self-review against the module-structure checklist plus EGYCASH's independent review. The
review approved the shape and raised **two blocking items**, both implemented before merge:

1. **Human-readable offer number** — every offer now carries an immutable, unique, searchable
   `JO-{YYYY}-{seq:6}` code (reusing the applicant-numbering pattern), used by HR instead of
   the ObjectId.
2. **Immutable accepted-revision snapshot** — acceptance freezes the exact terms into
   `acceptedSnapshot`, so later changes cannot alter what was actually accepted; Stage 5
   consumes the snapshot, not the mutable current terms.

The module boundary held (no `infrastructure` imports from `modules/`; cross-feature access to
Applicant / Interview via their `index.ts` barrels; platform Notifications and Scheduler via
`platform/*`); every transition is audited; the collection carries the `hr_` prefix; the new
`scheduledTasks` seam is generic and manifest-validated.

## 7. Lessons learned

1. **Reusing the numbering pattern paid off.** The `JO-` counter dropped straight onto the
   existing `hr_sequences` mechanism; the only wrinkle was reusing the shared Mongoose model
   without a second registration, solved with a model-cache guard.
2. **An explicit frozen snapshot beats "it can't change anyway."** Revision was already blocked
   post-acceptance, but a dedicated immutable `acceptedSnapshot` makes the Stage-5 contract
   unambiguous and defends against future edits — a small field that removes a class of
   "which terms were accepted?" questions.
3. **The scheduled-task seam generalized cleanly.** Automatic expiration needed a repeatable
   task; adding `ModuleManifest.scheduledTasks` (declared before the scheduler sync, prefix-
   validated) kept the module from reaching into platform bootstrap, and future modules inherit
   it for free.
4. **Notifications and audit stayed non-blocking seams**, consistent with every prior stage —
   an offer transition never fails because a notification or audit write did.
