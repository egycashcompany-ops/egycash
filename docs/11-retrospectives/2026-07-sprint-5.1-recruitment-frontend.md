# Sprint Retrospective — Sprint 5.1 (Release v0.13.0)

**Sprint:** 5.1 · **Capability:** HR / Recruitment — Frontend Foundation (Phase 1) · **PR:**
[#31](https://github.com/egycashcompany-ops/egycash/pull/31) · **Merged:** 2026-07-12 ·
**Outcome:** ✅ Delivered — reviewed by EGYCASH, **no blocking comments** (two backlog notes).

## 1. Sprint goal

Establish the **complete, reusable Recruitment web frontend foundation** — the shell, shared UI
kit, and platform integration every later screen builds on — **foundation-first, not
page-by-page**. Explicitly **no feature screen** (Applicants included) was to be built.

## 2. Delivered features

- **App shell & layout** — generic `AppShell` (sidebar + topbar), RTL-safe and responsive
  (persistent rail on desktop, off-canvas drawer on mobile), `Breadcrumbs`, `PageContainer`/
  `PageHeader`.
- **Theme** — light/dark/system (Tailwind `class` strategy + `ThemeProvider`, OS-reactive,
  persisted); brand token scale; **Arabic RTL** default via logical utilities.
- **Navigation & permission-aware routing** — `RequireAuth` + `RequirePermission` guards,
  `Can`/`useCan`/any-of/all-of role-based UI, 403/404 pages, recruitment module **lazy-loaded**
  (route-based code splitting).
- **Shared UI kit** (`shared/ui`) — DataTable, Pagination, SearchInput, FilterBar, BulkActions,
  Button, Field/Input/Textarea/Select/Checkbox/Form, Dialog, FileUpload, Badge/StatusBadge,
  Timeline, Card, Spinner/Skeleton, and Loading/Empty/Error/Success states.
- **API layer & data** — typed REST + multipart client (in-memory token per ADR-006, silent
  refresh, envelope unwrap, `buildQuery`), error-code → localized message mapping, query-key
  factory, and **TanStack Query with global error handling**; client toast store + `Toaster`;
  `NotificationBell`; top-level `ErrorBoundary`.
- **Recruitment module shell** — nav, permission-aware landing overview, and per-stage
  permission-gated placeholder routes for all seven stages.
- **i18n** — `ar` + `en` catalog with `{{param}}` interpolation.

## 5. Architecture review summary

Reviewed against Software Architecture §6 and ADR-013 plus EGYCASH's review — **approved, no
blocking comments**. State ownership held the ADR-013 line (TanStack Query = server state; Redux =
session/UI); the shared kit is wrapped once in `shared/ui` and imported via its barrel; RTL and
theme are systemic (logical utilities + class strategy), not per-component; permission gating is
UX-only with the server still authoritative.

## 6. Follow-ups recorded (EGYCASH review — non-blocking, for future sprints)

1. **Add Vitest + React Testing Library** before the Recruitment feature screens grow
   substantially, so component/interaction behaviour is covered by automated tests.
2. **Revisit adopting shadcn/ui later** if it provides measurable value — the current hand-rolled
   abstraction behind `shared/ui` is acceptable, and any migration stays localized to the kit.
3. **Live notifications inbox** — the bell + toast sink are in place; the server-backed inbox and
   Socket.IO badge land with that feature.

## 7. Lessons learned

1. **Foundation-first paid off immediately.** Building the shell, kit, routing, and data plumbing
   before any screen means the first feature (Applicants) is assembly, not invention — each stage
   route already resolves through the shell with breadcrumbs, guards, and states in place.
2. **Dependency-free primitives kept the build green in a constrained environment.** Honouring the
   §6 UI-kit *shape* (wrapped in `shared/ui`, tokenised, RTL-aware) without pulling in shadcn/Radix
   avoided an install risk while leaving a clean seam to adopt them later.
3. **Global error handling belongs at the data layer.** Centralising toast-on-failure in the
   Query/Mutation cache (with inline `ErrorState` for first loads) means feature screens get
   consistent error UX for free and rarely hand-roll error handling.
