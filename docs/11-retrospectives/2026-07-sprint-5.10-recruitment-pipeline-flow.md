## 1. Sprint goal

Make the finished seven-stage Recruitment module behave as a **continuous pipeline** — applicants
flow from stage to stage automatically, and can be withdrawn/restored from anywhere without losing
history — **while keeping the existing workflow, permissions, versioning, events and create/decide
APIs unchanged**.

## 2. Delivered features

- **Auto-appearing stage queues (derived read models).** `GET /hr/screenings/awaiting` (live
  applicants with no screening) and `GET /hr/interviews/awaiting` (accepted-screening applicants,
  still live, with no interview) surface applicants in the next stage automatically — computed
  server-side from the applicants' current state, **not** from fabricated placeholder records. Each
  drives an "Awaiting …" panel that opens the *existing* Start-screening / Schedule dialog. Both are
  read-only endpoints; the manual open/schedule/decide flows are untouched.
- **Optional interview committee.** `ScheduleInterview.interviewerIds` defaults to `[]`; an interview
  can be scheduled before a committee is assigned, with members added later via reassign-panel.
- **Withdraw / restore from any stage.** A shared `ApplicantLifecycleActions` control (Withdraw while
  `new`, Restore while `withdrawn`) on the applicant detail **and** every pre-hire stage detail page
  (Screening, Interviews, Job Offer). The post-hire stages deliberately do **not** expose it (the
  person is already an employee). `POST /hr/applicants/:id/restore` (version-checked, audited, emits
  `hr.applicant.restored`) returns the applicant to `new`; withdraw/restore invalidate the awaiting
  subtrees so the derived queues refresh immediately.
- **Exact-stage resume.** Because visibility is derived from the applicant's records, a restored
  applicant resumes from the exact stage they left — never restarting — and **no history is deleted
  or recreated** (screening decisions, interviews, offers, audit, timeline all intact).

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus several EGYCASH review rounds — **approved**. The key decision,
confirmed by EGYCASH: **derived "awaiting" queues, not auto-created records**, for both Screening and
Interviews — this keeps "automatically appears" + "avoid duplicate records" while leaving the
existing create/decide workflow and its semantics completely intact. The lifecycle control is a
single shared component reused across the pre-hire stage pages (no duplicated dialogs), and the
boundary (no withdraw/restore on post-hire stages) was an explicit, approved product decision. No new
runtime dependency; the only backend additions are two read-only endpoints and the restore
endpoint/event.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — the awaiting panels and `ApplicantLifecycleActions` are prime
   first-web-test candidates. Still the primary follow-up before declaring the module
   production-ready.
2. **Notifications on restore** — a restored applicant re-entering a queue could notify the owning
   recruiter (reuses the notifications seam) — a later nicety.
3. **Awaiting-queue pagination** — the queues are capped (limit 200); very large pipelines could page
   or filter by branch.

## 7. Lessons learned

1. **Derive visibility; don't materialize it.** Making the "awaiting" queues pure read models (state
   + existing records) is what let the pipeline feel automatic *and* keep every existing workflow,
   permission and API unchanged — and it made exact-stage resume fall out for free, because a
   restored applicant's records already place them correctly.
2. **Confirm the record-vs-view fork early.** The one genuine design fork (auto-create screening
   records vs. a derived queue) was surfaced for the reviewer rather than guessed; the derived
   approach avoided changing the manual create/decide semantics and their tests.
3. **One shared control beats N inline copies.** Extracting `ApplicantLifecycleActions` unified
   withdraw/restore across four pages and removed the applicant detail's duplicated dialogs — "from
   any stage" became a one-line placement rather than four implementations.
