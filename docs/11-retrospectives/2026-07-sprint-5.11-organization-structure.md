# Sprint Retrospective — Sprint 5.11 (Release v0.23.0)

**Sprint:** 5.11 · **Capability:** HR Foundation — Phase 1: Organization Structure · **PR:**
[#54](https://github.com/egycashcompany-ops/egycash/pull/54) · **Merged:** 2026-07-21 ·
**Outcome:** ✅ Delivered — reviewed by EGYCASH, **approved**.

## 1. Sprint goal

Build the **master organizational model** that every future module reuses — **Company, Branches,
Departments, Sections and the organization-wide Job Titles catalog** — on top of the existing
`platform/organization` backend (ADR-015), **without** rebuilding the hierarchy or adding
unnecessary APIs. Phase 1 only: **no** Job Positions, Job Requisitions, or Recruitment integration.

## 2. Delivered features

- **Organization admin (web).** A new lazy module at `/organization` with its own shell + nav and a
  cross-link to/from Recruitment:
  - **Company profile** — view + version-checked edit (name, legal name, tax number, commercial
    registry, fiscal-year start).
  - **Branch → Department → Section** — full CRUD (list/detail/create/edit/soft-delete), with parent
    pickers, optional manager, branch postal address, URL-synced filters, and the child-guard on
    delete surfaced to the user. All three units are **one generic `Unit*` implementation** configured
    per unit (`shared/unit-config.ts`) — mirroring the backend's `makeOrgUnitHandlers` factory, so the
    screens are "thin over one implementation."
  - **Job Titles** — full CRUD exposing every enriched field.
  - RBAC-gated (`RequirePermission`/`Can`), TanStack Query with the shared key factory, version-checked
    mutations, i18n (en + ar) + RTL. Ships as its own ~38 KB lazy chunk.
- **Enriched Job Titles (backend + contracts).** `jobGrade` (the only new required field) plus
  optional `description`, `salaryMin`, `salaryMax`, `requiredQualifications`, and numeric
  `requiredExperienceYears`. Salary-band coherence (`min ≤ max`) is enforced **twice**: the Zod schema
  refines when both bounds are present, and the service does a **merged-state** check on partial
  updates (a lone `salaryMax` is validated against the stored `salaryMin`) → `BUSINESS_RULE_VIOLATION`.
  Job Titles remain an **organization-wide catalog** — never tied to a Branch/Department/Section.
- **No new backend endpoints, permissions, or events.** The admin consumes the existing
  `/platform/{organization,branches,departments,sections,job-titles}` APIs and the existing RBAC
  catalog (verified: permission matrix unchanged via `--check`). No data migration (Job Titles were
  not seeded).

## 3. Test results

- **Backend** — integration coverage for the enriched Job Title catalog in `platform.spec.ts`: full
  create with every field, `jobGrade` required, create-time salary-band rejection, minimal
  (grade-only) create, the **merged-state** salary check on partial update, and search. All green.
- **Web** — no unit-test runner yet (backlog: Vitest + RTL). Gates were strict typecheck + repo lint
  + vite build, all green; organization is a lazy chunk.

## 4. CI results

- **Green.** One iteration was needed: two test expectations wrongly asserted **422** for *schema*
  validation failures (missing `jobGrade`, the create-time salary-band refine) — the API returns
  **400** for Zod validation and **422** only for business-rule violations. Fixed the expectations
  (the merged-state salary check keeps its 422); product behaviour never changed.

## 5. Architecture review summary

Reviewed against ADR-015 (organization model) and §6 — **approved**. Key decisions:

- **Enrich, don't rebuild.** The Company → Branch → Department → Section hierarchy already existed on
  the backend; Phase 1 enriched the Job Title catalog and added the missing web admin, rather than
  re-implementing the hierarchy.
- **One generic org-unit admin.** Branch/Department/Section are structurally identical, so their
  list/detail/form are a single configured implementation — the frontend mirror of the backend's
  shared org-unit factory. Company and the enriched Job Titles keep their own screens.
- **ADR-016 (now governing).** Job Positions & Job Requisitions are **OPTIONAL** for applicants; the
  **Talent Pool is first-class**; no future module may make the link required.

## 6. Phase status & deliberate deferrals

- **Phase 1 (Organization Structure) is complete and released** (v0.23.0).
- **Job Positions** and **Job Requisitions** are **intentionally deferred** to later phases (roadmap:
  Organization Structure → Job Positions → Job Requisitions → Recruitment integration).
- Acting-manager delegation windows are supported by the backend and left for a later admin polish
  (the form sets `managerId` but not the delegation window).

## 8. Lessons learned

1. **Check what already exists before building.** The whole hierarchy was already there; the sprint's
   value was enrichment + the admin UI, not a rebuild — which kept the change additive and the backend
   untouched (no new endpoints/permissions/events).
2. **Generic where the shape repeats.** Three identical units → one configured admin, echoing the
   backend factory. Less code, one place to change, and consistent behaviour.
3. **400 vs 422 is a contract, not a preference.** Schema (Zod) failures are 400; business-rule
   violations are 422. Tests must encode that distinction — the merged-state salary check is the only
   422 here because it is a genuine business rule, not a schema constraint.
4. **Write the invariant down where it governs.** Making the Talent-Pool rule an ADR (016), not just a
   code comment, gives future modules something to be reviewed against.
