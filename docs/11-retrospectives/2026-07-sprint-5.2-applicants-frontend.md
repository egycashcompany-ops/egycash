## 1. Sprint goal

Deliver the **Applicants** feature screens — the first Recruitment feature set — entirely on the
Phase 1 frontend foundation. **Applicants only**; no later stage.

## 2. Delivered features

- **List** — sortable/selectable `DataTable`; multi-filter bar (search + status / source / intake
  channel / identity-verification / duplicates-only / has-files); `Pagination`; **bulk withdraw**
  (reason dialog); **CSV export**; create entry point — all permission-gated. **Filters, search,
  sort and pagination are synchronized with the URL** (deep-linkable, back/forward aware).
- **Detail** — identity/contact/preferences/application read-out; **attachments** panel (upload
  with title+category, signed-URL download, remove); **verify-identity** and **withdraw** actions
  (version-checked).
- **Create / edit** — comprehensive manual-entry form on the shared primitives, server-authoritative
  validation surfaced in a summary, optimistic-concurrency-guarded edits.
- **OCR assist** — upload a National-ID image → extraction seam → apply fields with confidence
  bands; degrades to manual entry when no provider is wired (OQ-30).
- **Cross-module references** — Requisition/Branch use placeholder reference controls (disabled
  "coming soon" selector, or a read-only chip when supplied by context) — internal IDs are never
  editable fields.
- Feature `api/` layer + TanStack Query hooks; `ar` + `en` i18n. Added `getPage` + `downloadBlob`
  to the shared api-client.

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus EGYCASH's review — **approved**, with two changes folded in before
merge: (a) **URL-synced list state** (filters/search/sort/pagination in the query string), and
(b) **placeholder reference controls** replacing raw `jobRequisitionId`/`branchId` inputs. Both are
the correct architectural calls: URL as the single source of truth for list state, and a cross-module
reference seam that hides internal IDs and drops in a real picker later.

## 7. Lessons learned

1. **The foundation made the feature assembly, not invention.** List/detail/form came together from
   shared `DataTable`/`FilterBar`/`Pagination`/form primitives + the api-client, so effort went into
   applicant specifics (OCR flow, attachments, the rich form) rather than plumbing.
2. **URL-as-state is worth doing up front.** Deriving list filters/sort/paging from the query string
   gave deep links and back/forward for free and removed a class of "stale local state" bugs — it
   belongs in the shared list pattern for every later screen.
3. **Model missing cross-module dependencies as explicit seams.** A placeholder reference control
   that refuses to expose raw IDs — and gates creation until context is supplied — is both better UX
   and a clean integration point, far preferable to a temporary ID text box.
