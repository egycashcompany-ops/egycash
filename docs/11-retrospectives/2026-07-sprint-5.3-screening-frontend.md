## 1. Sprint goal

Deliver the **Initial Screening** feature screens — the second Recruitment feature set — on the
Phase 1 foundation, reusing the Applicants building blocks. **Screening only**; no later stage.

## 2. Delivered features

- **Queue** — sortable `DataTable` (status, notes, decided, created); filters (status +
  created-date range + an **applicant search-picker**); pagination — all **URL-synchronized**
  (deep-linkable, back/forward). **Start screening** dialog (pick a live applicant + optional first
  note).
- **Detail** — applicant link, **notes + decision timeline** (shared `Timeline`), an **add-note**
  form while `pending`, and the **Accept / Reject** workflow via a dialog (reason required to
  reject, OQ-32; optional to accept). Version-checked mutations.
- Feature `api/` layer + TanStack Query hooks against `/hr/screenings` (+ `/:id/notes`,
  `/:id/decide`); `ar` + `en` i18n; permission-gated (`screening.{view,create,edit,decide}`).

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus EGYCASH's review — **approved, no blocking comments**. The queue
reuses the URL-as-state list pattern established in Phase 2; the Accept/Reject workflow matches the
backend contract exactly (reason required to reject, version-checked); the applicant search-picker
reuses the Applicants API rather than inventing a screening search field.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — component/interaction tests (carried forward).
2. **Screening free-text search** — the backend screening list has no text field; a future backend
   `search` (applicant name/code) would let the queue search directly instead of via the
   applicant-picker seam.
3. **Applicant name in the queue** — the screening DTO carries `applicantCode`, not the name;
   surfacing the name would need a denormalized field or a lookup.

## 7. Lessons learned

1. **Reuse compounded.** The applicant search-picker, status-badge pattern, `Timeline`, dialogs, and
   the URL-as-state list pattern all came straight from Phase 1/2 — Screening was mostly wiring plus
   the decide/notes specifics.
2. **Map "search" to what the API offers, honestly.** Rather than fake a screening text search, the
   queue resolves an applicant (reusing the Applicants API) and scopes by `applicantId` — the
   correct behaviour for the available contract, with the limitation documented.
3. **Small, contract-exact workflow dialogs scale.** The decide dialog encodes exactly the backend
   rule (reason required to reject) and nothing more; keeping such actions thin and
   contract-faithful keeps the surface predictable across stages.
