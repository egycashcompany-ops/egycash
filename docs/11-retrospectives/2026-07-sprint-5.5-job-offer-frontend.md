## 1. Sprint goal

Deliver the **Job Offer** feature screens — the fourth Recruitment feature set — on the Phase 1
foundation, reusing the Applicants/Screening/Interviews building blocks. **Job Offer only**; no
later stage.

## 2. Delivered features

- **List** (`jobOffer.view`) — sortable `DataTable` (status, created); `OfferFilters` (a **free-text
  search** over offer number / applicant code — the offer list has a real `search` field — plus
  status + an active-only toggle); `Pagination` — all **URL-synchronized**. New-offer entry
  (`jobOffer.create`).
- **Create / revise** — the shared `OfferTermsForm` builds the versioned package (job title,
  department, branch, reporting manager, employment type, salary + currency, dynamic
  allowances/benefits, probation, start/validity dates, notes). Create picks an applicant; revise
  edits a draft/sent offer's terms (`jobOffer.edit`, version-checked, history preserved). Client
  checks cover the required fields + `validUntil > startDate`.
- **Detail** — the offer number, applicant link, status, the live package, the immutable
  **accepted-terms snapshot** and the **revision history**, plus the lifecycle actions **send /
  accept / reject / withdraw / revise** — each shown only in the states where it applies (draft·sent)
  and gated by permission (`jobOffer.{send,respond,withdraw,edit}`). Reason required to reject/withdraw.
- **References** reuse existing platform endpoints (**no new backend API**): the reporting manager
  via a single-select `ManagerPicker` over `/platform/users` (`user.view`); job title / department /
  branch via the org list endpoints. Raw ids never entered; degrade to a hint without the relevant
  `*.view`. Feature `api/` layer + TanStack Query hooks; `ar` + `en` i18n; permission-gated throughout.

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus EGYCASH's review — **approved**. The list reuses the URL-as-state
pattern; every write seeds the detail cache from the response and invalidates only the list subtree
(minimal invalidation); the lifecycle actions and revise/accept/reject/withdraw dialogs match the
backend contract exactly (reason required to reject/withdraw, version-checked); references reuse the
existing org/users endpoints rather than inventing a new API.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — component/interaction tests (carried forward).
2. **Reference directory coupling** — job title / department / branch / manager names resolve only
   with the relevant `*.view`; a future dedicated reference-directory read would remove the coupling.
3. **Offer expiry** — automatic expiry is a backend scheduled sweep; the UI reflects the resulting
   `expired` status but does not drive it.

## 7. Lessons learned

1. **Reuse held up under a bigger form.** The compensation package is the largest recruitment form
   since Applicants, yet it still sat entirely on the shared form primitives, dialogs, DataTable,
   filters, and URL-as-state pattern — the new surface was the terms/allowances/benefits editors and
   the offer lifecycle, nothing infrastructural.
2. **Pickers over raw ids, again.** The manager (user) and org references reuse existing endpoints
   through pickers/dropdowns that never expose internal ids — the same rule established for Applicants
   requisition/branch and Interviews interviewers, now applied to a four-reference form.
3. **Contract-exact lifecycle dialogs scale.** Send/accept/reject/withdraw each encode exactly the
   backend rule (reason required where the server requires it) and nothing more; keeping them thin
   and version-checked keeps the action surface predictable across stages.
