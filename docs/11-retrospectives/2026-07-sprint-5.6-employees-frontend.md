## 1. Sprint goal

Deliver the **Employees** feature screens — the fifth Recruitment feature set — on the Phase 1
foundation, reusing the prior phases' building blocks (including the Job Offer reference
infrastructure). **Employees only**; no later stage.

## 2. Delivered features

- **List** (`employee.view`) — sortable `DataTable` (employee `code`, hired, created);
  `EmployeeFilters` (a **free-text search** over employee number / applicant code + status);
  `Pagination` — all **URL-synchronized**. Hire entry (`employee.create`).
- **Hire / create** — the employment terms are **not** entered: they are copied server-side from the
  offer's immutable accepted snapshot. The page picks an **accepted offer** (an `OfferPicker`
  autocomplete reusing the Job Offer list API scoped to `status: accepted`) + an optional hiring
  date. The server enforces the full rule (accepted + snapshot + not already hired).
- **Detail** — the employee number, status, preserved references (applicant + accepted-offer links
  with revision), and the copied employment terms read-out — reusing the Job Offer `UserName` +
  reference hooks so org/manager names resolve from the same cache. Read-only stage.
- Feature `api/` layer + TanStack Query hooks; `ar` + `en` i18n; permission-gated
  (`employee.{view,create}`). **No new backend API.**

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus EGYCASH's review — **approved**. The list reuses the URL-as-state
pattern; the hire flow deliberately collects only the offer reference + optional hiring date (the
terms are server-copied from the accepted snapshot); the create write seeds the detail cache and
invalidates only the list subtree. Reference/offer lookups reuse the Job Offer API and reference
hooks rather than inventing a new endpoint; the employee record stays read-only, matching the
backend surface exactly.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — component/interaction tests (carried forward).
2. **Employee lifecycle** — leave/suspend/terminate transitions are not exposed by this stage; they
   belong to a future Employee module and would arrive with their own screens.
3. **Reference directory coupling** — org/manager names resolve only with the relevant `*.view`; a
   future dedicated reference-directory read would remove the coupling (shared across Job Offer +
   Employees now).

## 7. Lessons learned

1. **Cross-feature reuse paid off.** Employees imported the Job Offer `UserName` + reference hooks
   and the offer list API directly — the employment read-out and accepted-offer picker were pure
   wiring, no duplicated reference infrastructure.
2. **Model the UI to the backend's real surface.** The stage exposes only create-from-accepted-offer;
   the frontend adds no invented lifecycle actions, keeping the screen honest and read-only where the
   server is.
3. **Server-owned data stays server-owned.** The hire form never collects employment terms — they are
   copied from the immutable snapshot — so the UI can't drift from the accepted offer.
