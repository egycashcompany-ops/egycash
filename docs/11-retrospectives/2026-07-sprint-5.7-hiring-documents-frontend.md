## 1. Sprint goal

Deliver the **Hiring Documents** feature screens — the sixth Recruitment feature set — on the Phase 1
foundation, reusing the shared file-management infrastructure. **Hiring Documents only**; the
Electronic Employee File remains the final phase.

## 2. Delivered features

- **List** (`hiringDocuments.view`) — sortable `DataTable` (employee `code`, created); filters (a
  **free-text search** over employee number / applicant code + status); `Pagination` — all
  **URL-synchronized**. **Open document set** action (`hiringDocuments.create`) via an employee-search
  dialog.
- **Detail** — a **per-type checklist** merging the active type catalog with the uploaded documents:
  uploaded/missing (required flagged), **download** (signed-URL ticket), **version history**,
  **replace**, and **upload** for missing types (`hiringDocuments.upload`, PDF-only via the shared
  `FileUpload` + multipart). **Complete** (`hiringDocuments.complete`) is blocked — with the
  missing-required banner — until every required document is present; read-only once completed.
- Feature `api/` layer + TanStack Query hooks; version-checked mutations; `ar` + `en` i18n;
  permission-gated. The document-type catalog is consumed read-only. **No new backend API.**

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus EGYCASH's review — **approved**. Uploads/downloads reuse the
existing Files-service seams (`FileUpload`, multipart `upload`, the signed-URL download ticket from
Applicants attachments); the per-type checklist merges the read-only type catalog with the
aggregate's documents; the completion gate mirrors the backend rule exactly (blocked while
`missingRequired` is non-empty). Each write seeds the detail cache and invalidates only the list
subtree.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — component/interaction tests (carried forward).
2. **Document-type administration** — the catalog is consumed read-only; a management screen
   (create/edit/require types) is a later addition.
3. **Inline PDF preview** — documents open via a signed URL in a new tab; an embedded preview could

## 7. Lessons learned

1. **The file-management seam paid off.** `FileUpload`, the multipart `upload`, and the signed-URL
   download ticket built for Applicants attachments carried the whole hiring-documents upload/replace
   /download surface with no new infrastructure.
2. **Merge the catalog with the data.** Rendering the checklist from the active type catalog (not
   just the uploaded documents) makes missing/required states first-class and keeps the completion
   gate honest.
3. **Contract-exact gates, again.** The Complete action encodes exactly the backend rule and surfaces
   the missing-required list by name — thin, version-checked, and predictable like the other stages'
   actions.
