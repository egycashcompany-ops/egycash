## 1. Sprint goal

Improve the **Applicants intake** on the finished seven-stage Recruitment module: allow creating an
applicant **directly** (without a Job Request), and turn the National-ID OCR into a **dedicated,
reusable review experience** — visually separated from the Applicant form so it can be reused by
future modules (Employees, KYC, …).

## 2. Delivered features

- **Direct intake** — the Job Request is now **optional**. `jobRequisitionId` is **nullable
  end-to-end** (applicant → employee → employee-file all tolerate `null`); a supplied requisition is
  still validated (malformed ids rejected), and the link can be attached later when the Job Requests
  module lands. The `RefPickers` seam still never exposes raw ids.
- **Reusable National-ID OCR** (`apps/web/src/shared/national-id/`) — a module-agnostic flow that a
  host module drives by injecting an *extractor*: `NationalIdOcr` (two upload areas — **front +
  back** — read in one extraction pass) → a **dedicated `NationalIdReviewDialog`** showing **every**
  extracted field editable → **Confirm** hands back the reviewed data. Birth date / gender /
  governorate are **derived from the number** (`parseNationalId`, live, read-only), never OCR'd; the
  English name is seeded by transliterating the Arabic name. Applicants provide the binding
  (`ApplicantNationalIdOcr`) over the existing HR OCR endpoint and map the confirmed data into the
  form.
- **New identity fields** — `religion` + `nationalIdExpiry` (read from the card) added to the
  applicant contract, model, service, and mapper.
- The OCR extraction DTO was widened with the card fields; **no new backend endpoint**. `ar` + `en`
  i18n (a generic `nationalIdOcr.*` catalog).

## 5. Architecture review summary

Reviewed against §6/ADR-013 plus two EGYCASH review rounds — **approved**. The generic OCR flow
lives in `shared/` with **no HR coupling** (the extractor is injected), so it is genuinely reusable;
the review dialog is a dedicated modal that populates the host form **only on Confirm** (nothing is
saved and the form is untouched before then). Nullable `jobRequisitionId` propagates coherently
through the three aggregates that carry the reference, avoiding a latent hire-flow crash. No new
backend API; the OCR seam and File-Upload infrastructure were reused as-is.

## 6. Follow-ups recorded (non-blocking)

1. **Vitest + React Testing Library** — the shared `mapping`/`transliterate` helpers and the review
   dialog are pure/near-pure and prime candidates for the first web unit tests. **Still the primary
   follow-up before declaring the module production-ready.**
2. **Wire the reusable OCR into other flows** — Employees / KYC can adopt `NationalIdOcr` with their
   own extractor when those flows need ID capture.
3. **Real OCR provider** — the extraction seam is still the OQ-30 null stub; a real provider plugs in
   behind the same interface with no UI change.

## 7. Lessons learned

1. **Separate the reusable core from the module binding.** Splitting `NationalIdOcr` (generic UI +
   review) from `ApplicantNationalIdOcr` (the upload + endpoint binding) is what makes the component
   reusable — the generic half carries no module knowledge, and the binding is a few lines.
2. **A dedicated review step beats "the form is the review."** Moving the review into its own dialog
   made the OCR concern independent of the host form and clarified the flow (Extract → Review →
   Confirm → populate) — and it is exercisable even with the null OCR stub (the dialog still opens
   for manual entry).
3. **Relax constraints coherently, end-to-end.** Making `jobRequisitionId` optional meant following
   the reference through employee + employee-file so a direct-intake applicant stays hireable — a
   constraint change is only "done" when every consumer tolerates the new shape.
